
# 章节作业混合模式 - 课程JSON结构修改分析报告

## 问题分析

当前系统中，章节(Chapter)、作业(Homework)、和考试(Exam)是三个独立的模块：
- 章节：学习内容呈现（视频、文件、资源等）
- 作业：独立的练习题集合
- 考试：独立的测试题集合

**提议**：允许章节也可以作为一种作业/练习模式，实现混合模式。

---

## 解决方案概述

### 方案一：扩展章节(Chapter)结构 ⭐ 推荐方案

在现有chapter结构中添加`homeworkMode`配置，使章节可以同时具备学习和练习功能。

#### 优势：
- 最小化修改，向后兼容
- 充分利用现有结构
- 灵活性高

#### 劣势：
- 章节结构会变得复杂

---

### 现有路由

```
/student/course/[id]/section/[sectionId]
/student/course/[id]/homework/[homeworkId]
/student/course/[id]/exam/[exam]
```

### 新增路由（可选）

```
/student/course/[id]/section/[sectionId]/exercise  // 章节内练习
/student/course/[id]/chapter/[chapterId]/exercise  // 章节级练习
```

### 统一路由方案（推荐）

保持现有路由不变，在homework路由中支持type参数：

```
/student/course/[id]/homework/[homeworkId]?type=chapter_exercise
/student/course/[id]/homework/[homeworkId]?type=traditional_homework
/student/course/[id]/homework/[homeworkId]?type=exam
```

## 数据库/后端修改建议

### 新增表结构

```sql
-- 扩展courses表
ALTER TABLE courses ADD COLUMN mixed_mode_config JSON;

-- 新建chapter_exercises表
CREATE TABLE chapter_exercises (
  id VARCHAR(50) PRIMARY KEY,
  course_id VARCHAR(50),
  chapter_id VARCHAR(50),
  section_id VARCHAR(50),
  title VARCHAR(255),
  description TEXT,
  type VARCHAR(50),  -- 'chapter_exercise'
  difficulty VARCHAR(20),
  questions JSON,
  created_at TIMESTAMP,
  updated_at TIMESTAMP,
  FOREIGN KEY (course_id) REFERENCES courses(id),
  FOREIGN KEY (chapter_id) REFERENCES chapters(id),
  FOREIGN KEY (section_id) REFERENCES sections(id)
);

-- 新建student_chapter_exercises表
CREATE TABLE student_chapter_exercises (
  id VARCHAR(50) PRIMARY KEY,
  student_id VARCHAR(50),
  exercise_id VARCHAR(50),
  answers JSON,
  total_score DECIMAL(10,2),
  submitted_at TIMESTAMP,
  status VARCHAR(20),
  FOREIGN KEY (student_id) REFERENCES users(id),
  FOREIGN KEY (exercise_id) REFERENCES chapter_exercises(id)
);
```

### API端点

```
# 获取课程及其章节练习
GET /api/courses/:courseId?include=exercises

# 获取特定章节的练习
GET /api/courses/:courseId/chapters/:chapterId/exercises

# 提交章节练习
POST /api/courses/:courseId/exercises/:exerciseId/submit
Body: {
  studentId: string,
  answers: Answer[]
}

# 获取学生练习记录
GET /api/students/:studentId/chapter-exercises/:exerciseId
```

## 混合模式类型定义

```typescript
type HomeworkModeType = 'none' | 'homework' | 'mixed'

interface HomeworkModeConfig {
  enabled: boolean
  type: HomeworkModeType
  description?: string
  releaseDate?: string
  dueDate?: string
  pointsMultiplier?: number  // 1.0 = 标准权重
  settings?: {
    allowLateSubmission?: boolean
    latePenalty?: number
    randomOrder?: boolean
    immediateGrading?: boolean
    showCorrectAnswers?: boolean
    reviewAfterSubmit?: boolean
  }
}

interface SectionExercise {
  enabled: boolean
  practiceId: string
  questions: number
  optional: boolean
  displayMode: 'inline' | 'modal' | 'redirect'
}

type DisplayMode = 'inline' | 'modal' | 'redirect' | 'mixed'

interface ChapterExercise {
  id: string
  chapterId: string
  sectionId: string
  title: string
  description: string
  type: 'chapter_exercise'
  difficulty: 'easy' | 'medium' | 'hard'
  displayMode: DisplayMode
  questions: Question[]
  statistics?: {
    totalQuestions: number
    totalScore: number
    averageScore: number
  }
}
```

## 风险与注意事项

### ⚠️ 风险点
1. **数据重复**：章节练习数据可能同时存在于exercises和homework中
   - 解决：统一使用exercises存储，homework表中添加type字段区分

2. **显示模式复杂**：不同的displayMode可能导致UI逻辑复杂
   - 解决：使用条件渲染和组件化处理

3. **评分计算**：混合模式下的总分如何计算
   - 解决：使用pointsMultiplier乘数，在课程设置中配置权重

### 📋 最佳实践
1. 保持JSON结构简洁，复杂逻辑放在前端处理
2. 使用type字段明确区分不同类型的练习
3. 提供灵活的displayMode支持不同的UX设计
4. 实现增量迁移，不影响现有数据

---

## 总结

通过在现有Course JSON结构中添加：
- `homeworkMode` 配置（章节级别）
- `sectionExercise` 配置（节级别）
- `exercises` 数组（课程级别）

可以实现章节也可作为作业模式的混合系统，同时保持向后兼容性和最小化改动。
